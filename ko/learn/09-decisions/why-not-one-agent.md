# 왜 하나의 에이전트가 아닌가?

[HEAD Agent Core](../../README.md) / [학습](../README.md) / [결정](README.md) / 왜 하나의 에이전트가 아닌가?

## 문제

작고 국지적인 작업에는 하나의 대화로 충분할 수 있습니다. 하지만 긴 시간에 걸쳐 변하는 결과 안에서 사용자 방향, 폭넓은 근거, 전문 실행, 자체 검토를 모두 담아야 하면 신뢰하기 어려워집니다.

## 시도한 대안

계획, 실행, 검토를 하나의 장기 에이전트 컨텍스트 안에 둡니다. 하나의 책임자가 모든 이전 대화를 갖고 있어 인계 없이 질문에서 결과까지 갈 수 있으므로 단순해 보입니다.

## 관찰된 실패

**역사적 기록.** 초기 설계 자료는 전문 작업을 분리한 이유로 컨텍스트 증가, 역할 혼동, 순차적 병목을 설명했습니다. 또한 에이전트가 집중된 입력을 받고 정의된 출력을 낼 수 있으리라는 기대도 기록했습니다.

**운영 관찰.** 같은 컨텍스트가 결과를 만들고 그 적절성도 판단하면, 오래된 가정이 검토까지 이어질 수 있습니다. 긴 작업에서는 다음 결정에 필요한 근거와 무관한 이력이 경쟁하게 됩니다.

**일반화된 실패.** 범용 에이전트가 작은 릴리스를 계획하고 한 부분을 구현한 뒤 같은 불완전한 해석으로 검토합니다. 뒤이은 요청이 제약 하나를 바꾸지만, 이전 해석이 컨텍스트에서 더 두드러집니다. 검토는 바뀐 목표를 확인하는 대신 내부 일관성만 확인합니다.

## 현재 결정

전체 결과를 책임지는 하나의 상주 HEAD는 유지하되, 경계가 정해진 워커가 응집된 실행 결과를 맡도록 허용합니다. HEAD는 사용자 방향, 근거 선택, 통합, 최종 판단을 보유합니다. 워커는 결과를 만들고 확인하는 데 필요한 부분집합만 받습니다. 독립된 판단이 중요한 결과를 실질적으로 바꿀 수 있을 때는 독립 검토를 사용합니다.

이는 모든 작업을 위임하라는 규칙이 아닙니다. 위임의 조정 비용이 가치보다 클 때 HEAD가 직접 작업할 수 있습니다.

```text
사용자 방향 -> HEAD가 전체 결과 보유 -> 경계가 정해진 워커가 하나의 결과 생성
                              ^                         |
                              +---- 검증하고 통합
```

## 관련 이론

**관련 이론.** 직무 분리와 제어 평면 대 실행 평면 설계는 이 분할을 설명하는 데 도움이 됩니다. 이는 사후적 렌즈입니다. 역사적 기록은 관찰된 조정 문제를 뒷받침할 뿐, 이 이론들이 원래 설계 어휘였음을 입증하지는 않습니다.

## 현재 한계

HEAD는 여전히 판단이 집중되는 지점이며 사용자를 오해하거나 약한 근거를 선택할 수 있습니다. 이 분할은 인계 비용도 더합니다. 집중된 책임이나 독립된 반박이 결과를 개선할 때에만 정당화됩니다.

## 요점

하나의 컨텍스트를 완전한 제어 시스템으로 착각하지 마세요. 전체 결과의 판단은 결과에 책임지는 하나의 소유자에게 두고, 그것이 신뢰를 높일 때 경계가 정해진 실행이나 검토를 분리하세요.
