# 왜 런이며 런타임 상태가 아닌가?

[HEAD Agent Core](../../README.md) / [학습](../README.md) / [결정](README.md) / 왜 런이며 런타임 상태가 아닌가?

## 문제

오래 유지되는 작업은 중단, 압축 또는 다른 컨텍스트에서 재개될 수 있습니다. 시스템은 생성된 인계나 런타임 포인터가 원래 합의를 재정의하지 않으면서 원래 합의를 보존해야 합니다.

## 시도한 대안

현재 작업 표식, 스냅샷, 생성된 요약, 활성 워커 상태를 다음에 무엇을 해야 하는지에 대한 권위 있는 설명으로 취급합니다. 한 기록이 없을 때 복구하도록 탐색 및 대체 메커니즘을 추가합니다.

## 관찰된 실패

**역사적 기록.** 이전 설계에는 상태 파일, 작업 패키지, 복구 메커니즘이 누적되었습니다. 현재 복구 계약은 대체 경로를 고르는 방식을 작고 고정된 정본으로 의도적으로 대체했습니다.

**운영 관찰.** 생성된 진행 기록은 검색에는 유용하지만 제약을 빠뜨리거나 미해결 작업을 완료로 압축하거나 오래된 해석을 보존할 수 있습니다. 대체 수단은 잘못된 작업 합의를 선택하면서도 복구가 견고해 보이게 할 수 있습니다.

**일반화된 실패.** 중단 후 요약은 작업이 거의 끝났다고 말하지만 사용자 제약 하나를 빠뜨립니다. 복구는 요약을 권위로 취급하고 축소된 목표를 검증한 뒤, 잘못된 방향으로 확신하며 계속합니다.

## 현재 결정

오래 유지되는 작업에서는 목적, 범위, 성공 조건, 결정, 구조, 현재 상황, 다음 결과, 근거 위치, 체크리스트를 담은 안정적인 런을 사용자-HEAD 합의로 보존합니다. 세션 정체성은 분리합니다. 생성된 진행과 이력은 취약한 검색 보조물로 취급합니다. 복구는 고정된 정본 기록에서 시작하고 필요할 때만 뒷받침하는 슬라이스나 근거를 명시적으로 검색합니다.

```text
사용자-HEAD 합의 -> 오래 유지되는 런 -> 복구와 계속
검증된 갱신 -> 오래 유지되는 런
취약한 런타임 기록 -> 검색 전용, 권위가 아님
```

## 관련 이론

**관련 이론.** 단일 진실 소스 설계, 이벤트 소싱 경계, 불변 합의 사고는 파생 상태가 오래 유지되는 의도를 덮어쓰지 않아야 하는 이유를 설명합니다. 이는 개념적 대응이며 시스템이 특정 형식 지속성 모델 하나를 구현한다는 주장이 아닙니다.

## 현재 한계

런도 잘못 쓰이거나 오래되거나 오해될 수 있습니다. 고정 기록은 누락된 외부 근거를 복구하지 못하며, 큰 런을 재개하는 데도 판단이 필요합니다. 합의한 방향을 바꾸는 권한은 사용자에게 남아 있습니다.

## 요점

작업 합의는 모델이 관리하는 상태 밖에서 정본으로 유지하세요. 런타임 기록은 컨텍스트를 찾는 데만 쓰고 목표를 조용히 다시 쓰는 데 쓰지 마세요.
