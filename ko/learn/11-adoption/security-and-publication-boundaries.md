# 보안 및 공개 경계

[HEAD Agent Core](../../../README.md) (영문) / [Learn](../../../learn/README.md) (영문) / [도입](README.md) / 보안 및 공개 경계

## 학습 목표

인가, 민감한 컨텍스트, 프로덕션 변경, 외부 커뮤니케이션을 에이전트 아키텍처가 암시하는 역량이 아니라 프로젝트가 통제하는 경계로 다룹니다.

## 핵심 주장

HEAD 조정은 행동을 인가하지 않습니다. 프로젝트와 인간 소유자가 자격 증명, 접근, 변경 정책, 외부 커뮤니케이션 승인, 공개 가능한 것을 정의합니다.

| 경계 | 필요한 통제 |
| --- | --- |
| 자격 증명 및 인증된 접근 | 프로젝트 통제 시스템에 둔다. 엄격히 필요하고 인가된 경우가 아니면 Shared Core, 공개 예시, 워커 브리프에 두지 않는다. |
| 비공개 프로젝트 컨텍스트 | 프로젝트 규칙 아래 관련 정본 소스만 검색한다. 공개 문서나 넓은 워커 컨텍스트에 복사하지 않는다. |
| 프로덕션 또는 파괴적 변경 | 명시적 프로젝트 인가와 적절한 승인 경계를 요구한다. 이용 가능한 도구가 권한을 부여하지 않는다. |
| 외부 메시지 및 공개 | 전송 또는 공개를 별도 승인 경계로 취급하고, 중요한 내용과 대상 영향에는 인간 방향을 둔다. |
| 공개 문서 | 비공개 제품 세부, 서비스 또는 저장소 식별 정보, 스키마, 내부 경로, 측정값, 프롬프트 본문, 운영 데이터, 현재 작업 상태를 제거한다. |

## 설계 대응

휴대 가능한 계층에는 공유 안전 원칙을 두고, 프로젝트 정책과 통합 계약을 통해 프로젝트별 인가를 강제합니다. 시스템이 도구를 노출할 수 있다는 이유만이 아니라 런타임 통제가 실제로 안전 경계를 강화할 때 호출 가능 인터페이스를 사용합니다.

## 이것이 보장하지 않는 것

에이전트는 정책을 오독할 수 있고, 프로젝트는 부적절한 경계를 구성할 수 있으며, 인간은 안전하지 않은 행동을 승인할 수 있습니다. 통제는 특정 위험을 줄이지만 책임이나 검토 필요를 없애지는 않습니다.

## 흔한 오해

“읽기 전용” 또는 “내부”가 자동으로 정보를 공유해도 안전하게 만든다. 민감성은 개별 호출이 변경을 일으키는지뿐 아니라 데이터, 집계, 대상, 프로젝트 정책에 달려 있습니다.

## 요점

접근, 변경, 공개, 중요한 커뮤니케이션은 프로젝트와 인간의 통제 아래 두세요. 아키텍처는 책임을 설명할 뿐 권한을 부여하지 않습니다.
