# 공유 계층과 프로젝트 계층 분리

[HEAD Agent Core](../../README.md) / [학습](../README.md) / [도입](README.md) / 공유 계층과 프로젝트 계층 분리

## 학습 목표

재사용 가능한 행동은 휴대 가능하게 유지하면서 프로젝트 사실, 권위, 위험 결정은 이를 소유한 사람에게 두는 법을 배웁니다.

## 핵심 주장

공유 지침은 HEAD가 어떻게 추론하고 작업을 조합하는지 말합니다. 프로젝트 계층은 한 환경에서 무엇이 참이고, 허용되며, 권위 있는지 말합니다. 이를 섞으면 둘 다 신뢰성이 떨어집니다.

| 계층 | 소유하는 것 | 소유하지 않는 것 |
| --- | --- | --- |
| Shared Core | 휴대 가능한 책임, 컨텍스트, 작업 모델, 정본, 검증 원칙 | 도메인 정책, 저장소 세부, 현재 작업, 자격 증명 |
| 공유 Skills, MCP, Agents | 재사용 절차, 호출 가능 계약 또는 역할 경계 | 프로젝트 라우팅, 지역 승인, 전문 사실, 비공개 통합 |
| 프로젝트 Core 및 색인 | 지역 권위, 제약, 정본 소스, 검색 경로 | 다시 쓴 보편 Core |
| 프로젝트 확장 | 도메인 작업 흐름, 통합, 오버레이, 전문가 | 무관한 공유 행동 |
| 사용자 | 제품, 정책, 아키텍처, 비용, 위험, 작업 흐름에 관한 중요한 방향과 최종 선택 | 안전하게 위임된 일상 실행 세부 |

## 공개 경계

트리거, 권한 경계, 절차 또는 계약, 예상 결과가 프로젝트 사실을 제거한 뒤에도 의미가 있을 때만 공유 아티팩트를 공개하세요. 그렇지 않으면 프로젝트 소유로 둡니다. 공개 참조는 비공개 지시, 프롬프트 본문, 스키마, 경로, 자격 증명, 운영 데이터, 현재 작업 상태를 재현하지 않고 경계를 설명해야 합니다.

## 설계 대응

색인은 컨텍스트 덤프보다 나을 때가 많습니다. 질문이 중요할 때 HEAD를 현재 정본 소스로 안내합니다. 이는 비공개 지식을 프로젝트 소유자가 통제하게 하고 오래된 복제를 줄입니다.

## 관련 이론

이는 관심사 분리와 경계가 정해진 컨텍스트의 사후적 적용입니다. 이 생각들은 분할을 명료하게 하지만 특정 분할이 완전하거나 올바름을 입증하지는 않습니다.

## 흔한 오해

“공유”가 “공개해도 안전함”을 뜻하지는 않습니다. 재사용성과 공개 안전성은 별도 테스트입니다. 일반적으로 보이는 지시도 비공개 운영 모델이나 역량을 드러낼 수 있습니다.

## 요점

휴대 가능한 행동은 공유 계층에 두고 지역 사실, 권위, 공개 결정은 프로젝트와 인간 소유자에게 두세요.
