# HEAD가 필요한 때

[HEAD Agent Core](../../README.md) / [학습](../README.md) / [도입](README.md) / HEAD가 필요한 때

## 학습 목표

명시적 결과 소유자가 작업을 개선하는 때와 직접 대화 또는 일반 작업 흐름으로 충분한 때를 알아봅니다.

## 핵심 주장

전체 결과를 보존하고, 의존성을 해결하고, 결과를 검증하는 일이 하나의 명확한 행동을 수행하는 일보다 어려울 때 HEAD를 사용하세요. LLM을 사용할 수 있다는 이유만으로 추가하지 마세요.

| 작업 형태 | 유용한 대응 |
| --- | --- |
| 영향이 작고 즉시 확인 가능한 하나의 명확한 행동 | 직접 작업한다. 런이나 워커를 뜻하지 않는다. |
| 여러 의존성, 인계 또는 중단 위험이 있는 작업 | HEAD가 작업 모델을 만들고 합의를 보존하게 한다. |
| 별도 실행이 이로운 경계가 정해진 결과 | 하나의 응집된 결과를 위임하고 검증한다. |
| 두 번째 판단이 결정을 바꿀 수 있는 중요한 결과 | 독립 검토를 추가한다. |
| 중요한 제품, 정책, 아키텍처, 비용, 위험, 작업 흐름 선택 | 인간 결정 게이트를 명시적으로 유지한다. |

## 부적합한 사용

HEAD는 단순한 편집을 계획 연습으로 바꾸거나, 생성된 확신을 근거 대신 사용하거나, 에이전트 작업 흐름 뒤에 미해결 인간 결정을 숨길 이유가 아닙니다. 군집도 나쁜 기본값입니다. 소유자가 많을수록 컨텍스트, 조정, 조율 작업도 늘어납니다.

## 실용적 테스트

네 가지 질문을 하세요:

1. 끝에 참이어야 할 결과는 무엇인가?
2. 작업을 나중에 이어가면 무엇이 잊히거나, 추측되거나, 바뀔 수 있는가?
3. 별도 소유자에게 응집된 결과와 관찰 가능한 완료 근거가 있는가?
4. 어떤 선택이 결과를 실질적으로 바꾸므로 인간 소유로 남아야 하는가?

답이 단순하고 안정적이면 직접 작업이 보통 낫습니다. 연속성, 조정 또는 근거 문제를 보이면 그 문제를 다루는 계층만 추가하세요.

## 흔한 오해

“구조가 많을수록 신뢰성이 높다.” 구조는 결정과 근거를 검사하기 쉽게 할 수 있지만, 누락된 사실을 참으로 만들거나 나쁜 판단을 올바르게 만들 수는 없습니다.

## 요점

HEAD는 기본 의식이 아니라 책임과 검증이 필요한 복잡성을 위해 도입하세요.
