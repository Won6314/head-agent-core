# Core: 이식 가능한 원칙, 항상 존재함

[HEAD Agent Core](../../README.md) / [학습](../README.md) / [구성 요소](README.md) / Core

## 학습 목표

Core가 프로젝트 사실이나 상세 작업 지시가 아니라 이식 가능한 소유권과 추론 원칙을 담는 이유를 이해합니다.

## Core가 소유하는 것

Core는 HEAD에 전체 결과를 유지하는 안정적인 방법을 제공합니다. 즉, 컨텍스트를 의도적으로 조합하고, 작업 모델을 만들며, 일관된 소유권을 보존하고, 결정 소유권을 존중하며, 근거에서 확신을 구축하게 합니다. 이 원칙들은 HEAD가 어떤 프로젝트나 작업을 수행하는지 알기 전에도 유용합니다.

이 지침은 항상 로드되므로 작고 폭넓게 적용 가능하며 로컬 이름, 시스템, 정책으로부터 독립적이어야 합니다. 특정 프로젝트가 무엇을 결정해야 하는지가 아니라 HEAD가 작업을 어떻게 추론해야 하는지를 말합니다.

## 여기에 속하지 않는 것

| 내용이 답하는 질문... | 속해야 하는 곳... |
| --- | --- |
| 이 프로젝트에서 무엇이 일어날 수 있는가? | 프로젝트 Core 또는 프로젝트 정책 |
| 권위 있는 근거는 어디에 있는가? | 프로젝트 컨텍스트와 그 인덱스 |
| 조건부 워크플로를 어떻게 실행하는가? | Skill |
| 어떤 호출 가능한 조치를 사용할 수 있는가? | MCP |
| 누가 경계가 정해진 결과를 소유할 수 있는가? | Agent 계약 |
| 활성 사용자 합의는 무엇인가? | 세션 및 런 정본 |

이 분리는 중요도의 위계가 아닙니다. 로컬 안전 규칙은 작업에 결정적일 수 있습니다. 단지 공유 행동을 정의하기에는 이식성이 충분하지 않을 뿐입니다.

## 배치 검사

프로젝트별 사실, 경로, 도구, 전문 이름을 모두 제거합니다. 그 지침이 여전히 프로젝트 전반에서 건전한 소유권과 추론을 설명한다면 Core일 수 있습니다. 그 참됨이 특정 환경이나 결정에 의존한다면 프로젝트 소유자나 온디맨드 절차가 필요합니다.
