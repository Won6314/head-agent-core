# 하나의 세션에서 여러 역할로

[HEAD Agent Core (영문)](../../../README.md) / [학습 과정 (영문)](../../../learn/README.md) / [기원](README.md) / 하나의 세션에서 여러 역할로

## 학습 목표

왜 모든 종류의 작업을 담는 자연스러운 그릇으로 하나의 긴 LLM 대화를 더 이상 취급하지 않게 되었는지 이해한다.

## 출발점

처음에는 가능한 한 단순하게 구성했다. 하나의 대화, 하나의 모델, 같은 컨텍스트 안의 모든 작업이었다. 조사, 계획, 구현, 검토, 문서화가 모두 차례대로 이루어졌다.

프로젝트와 요청의 규모가 작을 때는 이 방식이 통했다. 모델은 관련 자료를 볼 수 있었고, 사용자는 최근 결정을 기억할 수 있었으며, 실수가 멀리까지 전파되는 일도 드물었다.

작업이 커지면서 서로 다른 세 가지 문제가 얽히기 시작했다.

### 컨텍스트 혼합

여러 목적으로 사용된 대화에는 서로 양립하기 어려운 작업 자료가 쌓였다. 조사 메모는 구현 세부 사항 옆에 남았다. 실패한 시도는 채택된 결정 옆에 남았다. 이전 요청에 맞춰진 컨텍스트 안으로 새로운 요청이 들어왔다.

문제는 컨텍스트 창의 용량만이 아니었다. 한계에 도달하기 전부터 모델은 현재 권위를 갖는 정보와 과거의 잔여물을 구분해야 했다. 자료가 늘수록 오래된 가정을 따르거나, 무관한 세부 사항에 지나친 비중을 두거나, 다음 결정을 실제로 지배하는 몇 줄을 놓칠 가능성도 커졌다.

### 역할 혼동

작업의 종류에 따라 필요한 결정권이 다르다. 조사자는 근거를 찾아낼 수 있다. 개발자는 국소적인 구현을 선택할 수 있다. 검토자는 결과를 몰래 재설계하지 않으면서 그 결과에 이의를 제기해야 한다. 이 역할들이 구분되지 않은 하나의 세션에 모두 있으면, 같은 모델이 결정을 제안하고 구현한 뒤 자기 가정을 스스로 승인할 수 있었다.

문제는 하나의 모델이 그런 행동을 수행할 능력이 없다는 것이 아니었다. 어떤 관점이 활성화되어 있는지, 어떤 주장이 권위를 갖는지, 누가 목표를 변경할 수 있는지를 워크플로가 보존하지 못한다는 것이 문제였다.

### 순차 처리 병목

하나의 세션은 한 번에 하나의 대화 흐름만 진행할 수 있으므로 독립적인 작업도 무관한 작업 뒤에서 기다려야 했다. 원칙적으로는 여러 작업을 동시에 수행할 수 있었다. 실제로는 명시적 소유권 없이 병렬 처리하면 편집 충돌과 해석의 분기를 일으킬 위험이 있었다.

## 첫 번째 분리

첫 번째 아키텍처는 장기 실행되는 역할 에이전트들을 분리했다. 각 역할은 격리된 컨텍스트, 집중된 지침 세트, 정의된 입력, 예상 출력을 받았다. 조정 역할인 HEAD가 단계를 계획하고 산출물을 연결했다.

```text
사용자
  |
  v
HEAD 조정자
  |-- 조사 역할
  |-- 계획 역할
  |-- 구현 역할
  `-- 검증 역할
```

이렇게 하자 세 가지가 즉시 개선되었다.

- 서로 무관한 실행 세부 사항이 같은 대화 컨텍스트를 차지하지 않게 되었다.
- 의존 관계가 명확할 때 독립 역할들이 병렬로 진행할 수 있었다.
- 전체 과정을 다시 시작하는 대신 역할 또는 단계 경계에서 실패한 작업을 반복할 수 있었다.

동시에 다음 세대의 문제도 생겼다. 역할 에이전트들은 서로 다른 방향으로 표류할 수 있었다. 이들의 상태를 전달하고 복구해야 했다. 에이전트가 늘수록 라우팅, 모니터링, 동기화 작업도 늘어났다. 시스템은 구조를 얻었지만, 구조만으로 일관된 소유권이 보장되지는 않았다.

## 첫 번째 분리에서 배운 것

오래 남은 통찰은 "에이전트는 많을수록 좋다"가 아니었다. 범위는 더 좁았다.

> 서로 다른 결정 범위에는 명시적인 소유자와 서로 다른 컨텍스트 경계가 필요하다.

초기 설계는 역할을 경계로 사용했다. 현재 설계는 그 발상을 더 다듬는다. HEAD는 전체 결과의 유일한 소유자로 남고, 에이전트는 일관되고 경계가 정해진 하나의 결과를 일시적으로 소유한다.

## 관련 이론: 회고적 관점

이 초기 분리는 여러 확립된 개념과 닮아 있다.

- 단일 책임: 하나의 역할에는 하나의 변경 이유만 있어야 하기 때문이다.
- 경계가 정해진 컨텍스트: 같은 용어와 사실이 모든 곳에서 모든 것을 의미할 필요는 없기 때문이다.
- 계층적 계획: 상위 목표를 하위의 실행 가능한 결과로 구체화하기 때문이다.
- 방향성 비순환 스케줄링: 실제 의존 관계는 순서를 유지하면서 독립적인 결과는 함께 진행할 수 있기 때문이다.

이 개념들은 지금의 관점에서 유용한 설명이다. 역사적 기록이 보여 주는 것은 컨텍스트 혼합, 역할 혼동, 순차 작업에 대한 운영상의 대응이지, 처음부터 그런 이론을 형식적으로 구현하려고 아키텍처를 시작했다는 사실이 아니다.

## 흔한 오해

이 분리의 주된 목적은 인간 회사를 흉내 내거나 모델에 인격을 부여하는 것이 아니었다. 중요한 단위는 성격이 아니라 책임이었다. 역할은 하나의 결과를 중심으로 컨텍스트와 결정 권한을 제한하기 위해 존재했다.

## 핵심 정리

서로 무관한 컨텍스트, 양립하기 어려운 결정권, 독립적인 작업이 서로 방해하기 전까지는 하나의 세션으로 충분하다. 첫 번째 다중 에이전트 설계는 그런 증상을 해결했지만, 에이전트 수가 일관된 소유권과 같지는 않다는 사실도 드러냈다.
