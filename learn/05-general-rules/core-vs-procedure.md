# Core Versus Procedure

[HEAD Agent Core](../../README.md) / [Learn](../README.md) / [General Rules](README.md) / Core Versus Procedure

## Learning Objective

Separate portable principles that belong in the shared Core from detailed, conditional procedures that belong in Skills.

## Different Jobs Need Different Delivery

| Layer | Carries | Use when |
| --- | --- | --- |
| Shared Core | Stable ownership and reasoning principles | Every HEAD decision needs the same foundation |
| Project layer | Project-owned facts, policy, and authority pointers | The work depends on that project's reality |
| Skill | Detailed conditional procedure and domain usage knowledge | A particular task, tool, or situation calls for it |

Always-loaded guidance must stay small and portable. A detailed procedure is valuable when it gives a reliable sequence for a particular condition, but loading every procedure into every decision makes the Core harder to understand and easier to misapply.

## A Placement Test

Ask whether the guidance remains true across projects and whether it is needed for ordinary reasoning. If yes, it may belong in Core. If it depends on a tool, environment, special risk, or task shape, keep it conditional and retrieve it when relevant. If it defines what a particular project may do, its owner is the project layer, not shared Core.

## Common Misunderstanding

Core versus Skill is not a quality hierarchy. A Skill can be more detailed and more important for a particular task. The difference is scope, ownership, and loading time.

## Takeaway

Load stable principles everywhere; retrieve detailed procedures and project facts only at the decision boundary that needs them.
