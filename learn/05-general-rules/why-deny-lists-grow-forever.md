# Why Deny Lists Grow Forever

[HEAD Agent Core](../../README.md) / [Learn](../README.md) / [General Rules](README.md) / Why Deny Lists Grow Forever

## Learning Objective

Recognize why adding one prohibition after each failure eventually creates guidance that is harder to apply and still incomplete.

## The Accumulation Pattern

A failure occurs. A new rule forbids the visible mistake. Another case appears, so the rule gains an exception. Over time, guidance becomes a record of remembered incidents rather than a model for choosing well in an unfamiliar situation.

```text
failure -> prohibition -> exception -> exception to the exception
        -> longer guidance with new gaps still possible
```

The issue is not that prohibitions are always wrong. A hard safety boundary may need a direct prohibition. The issue is using a prohibition list as the primary reasoning system for portable work.

## What Gets Lost

Long lists make priority unclear. Two rules can appear to conflict. An owner may follow literal wording while missing the protected outcome. New situations remain uncovered because the list only names cases already seen.

## Design Response

Extract the reason a boundary matters: preserve authority, inspect evidence before acting, keep the outcome bounded, or do not invent a missing policy choice. Then use specific procedures only where a condition, tool, or risk requires them.

## Common Misunderstanding

Generative guidance is not vague encouragement. A useful principle names an owner, a reasoning obligation, and an observable boundary that can be checked.

## Takeaway

Keep direct prohibitions for non-negotiable boundaries; use principles to cover the open-ended cases between them.
